# 上下文分层原则 — 完整规范

## 核心原则

> 按信息的**使用频率**和**稳定性**分层管理，每层只放自己该放的东西。

信息稳定性越高、使用频率越高 → 越靠近常驻层。
信息变化越频繁、使用越情境化 → 越靠近运行时注入层或系统层。

---

## 层 1：常驻层（Permanent Context）

**定义**：每次会话都必须成立的内容。

**放什么**：
- Agent 身份定义（角色、目标、语气规范）
- 项目级约定（技术栈声明、输出格式强制要求）
- 绝对禁止项（never do 清单）
- 关键 Harness 定义（验收标准锚点）

**不放什么**：
- 领域知识（→ Skills）
- 操作手册、参考文档（→ Skills）
- 动态数据（→ 运行时注入）

**质量标准**：短、硬、可执行。每一条都是约束，不是说明。

**反例**：
```
❌ 你是一个专业的代码审查助手，你需要仔细检查每一行代码，
   确保符合我们团队的规范...（300字说明）
```
```
✅ 代码审查助手。输出格式：[严重|警告|建议] 文件:行号 问题描述。
   禁止：输出未经验证的修复代码。
```

---

## 层 2：按需加载层（On-Demand Skills）

**定义**：描述符（name + description）常驻，完整内容触发时注入。

**放什么**：
- 领域知识（API 规范、业务规则、技术文档摘要）
- 操作流程（标准 SOP、检查清单）
- 专业判断框架（评分模型、决策树）

**机制**：
- 每个 Skill 的 description（~100 tokens）始终在 context 中
- 触发条件满足时，SKILL.md 完整内容（≤5000 tokens）注入
- 未触发时零 token 占用

**设计要求**：
- 每个 Skill 单一职责，不交叉
- 触发条件必须明确写在 description 中，避免模型猜测
- 详细内容放 `references/`，SKILL.md 只保留核心指令

---

## 层 3：运行时注入层（Runtime Injection）

**定义**：每轮对话按需动态拼入的信息。

**放什么**：
- 当前时间戳
- 会话渠道 ID、用户 ID
- 用户当前偏好/设置
- 当前任务状态快照（来自外部状态存储）
- 相关 RAG 检索结果

**注入方式**：
```python
# 每轮构建 user message 前拼入
runtime_context = f"""
<runtime>
current_time: {datetime.now().isoformat()}
user_id: {user.id}
channel: {channel_id}
task_state: {json.dumps(task_state_snapshot)}
</runtime>
"""
```

**约束**：每轮运行时注入 ≤200 tokens，超出则重新设计状态传递机制。

---

## 层 4：记忆层（Memory）

**定义**：跨会话积累的经验，不直接进 system prompt，按需读取。

**四类记忆**：

| 类型 | 内容 | 存储位置 | 读取时机 |
|------|------|----------|----------|
| 工作记忆 | 当前任务中间状态 | 文件系统（task_state.json） | 每轮 |
| 程序性记忆 | 操作流程、最佳实践 | MEMORY.md → Skills | 相关任务触发 |
| 情景记忆 | 具体事件、决策历史 | memory/episodes/ | RAG 检索 |
| 语义记忆 | 领域概念、项目知识 | memory/knowledge/ | RAG 检索 |

**整合策略**：
- 监控 token 使用，超过 80% 阈值自动触发摘要
- 摘要保留决策、错误和关键约束，丢弃过程细节
- 错误经验立即写入，成功模式累积 3 次后提炼写入

---

## 层 5：系统层（System Execution）

**定义**：完全不进上下文，由代码/Hook 执行的确定性逻辑。

**放什么**：
- 格式校验（JSON schema 验证）
- 权限检查（用户是否有操作权限）
- 幂等性保证（防重复执行）
- 速率限制
- 强制日志记录

**原则**：能用代码判断的，绝不让 LLM 判断。LLM 只处理需要语义理解的部分。

---

## Context Rot 防治完整策略

```
信号监控 → 分层审查 → 重构执行

信号：
- System prompt token 计数 > 2000
- 对话轮次 > 20 且质量下降
- 同一规则出现 2 次以上

分层审查：
1. 常驻层：是否混入知识类内容？→ 提取到 Skills
2. 技能层：是否有未 Skill 化的领域知识？→ 拆分新 Skill
3. 运行时：是否有写死的动态信息？→ 改为注入
4. 记忆层：是否有压缩机制？→ 配置阈值触发

重构执行：
- 常驻层 > 500 tokens：提取为 Skill
- 重复规则：合并为单一强约束
- 动态数据：移至运行时注入
```

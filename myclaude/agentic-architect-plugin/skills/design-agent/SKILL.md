# /design-agent — Agent 架构设计

接收业务场景描述，输出结构化的 Agent 架构设计文档。

## 执行前置条件

在生成架构设计前，必须完成以下分析：

1. **Harness 可行性评估**：用任务清晰度 × 验证自动化程度四象限定位场景
2. **上下文分层设计**：参照 context-design skill 原则
3. **工具链设计**：参照 tool-design skill 的 ACI 原则
4. **反模式预检**：对照 anti-patterns skill 的 8 条清单，标记潜在风险

## 思维链（执行前强制推演）

```
Step 1 - 业务诊断：
  - 真实业务约束是什么？（不是表面症状）
  - 谁受益？多频繁？不解决的成本？

Step 2 - Harness 定位：
  - 任务清晰度：目标是否明确、可操作？(1-5分)
  - 验证自动化程度：结果是否可自动验证？(1-5分)
  - 象限位置：右上（适合 Agent）/ 其他（需要先解决前提条件）

Step 3 - 架构模式选择：
  加载 references/architecture-decision-tree.md，使用决策树系统化选型，不得凭直觉猜测。
  - 单 Agent Workflow DAG（首选，步骤可在设计时确定）
  - 单 Agent ReAct Loop（步骤动态，需 LLM 推理决定下一步）
  - 多 Agent（需满足并行加速/专业分工/故障隔离之一，否则禁止引入）
  - 人机协同（验证无法自动化，或法规要求人工审批节点）

Step 4 - 工具链设计：
  - 每个工具：单一职责，ACI 原则，错误可恢复
  - 总数约束：≤10 个

Step 5 - 风险预判：
  - 对照 8 条反模式，标记哪些需要提前防范
```

## 输出质量要求

- 结论先于推理，每节直接给出判断
- 架构选型必须说明 tradeoff，引用决策树路径
- 所有假设标记 [ASSUMPTION]
- 状态机必须用 Mermaid.js，并通过 6 项质量清单
- 工具链必须过 ACI 检查
- Harness 定义必须可量化、可自动检测

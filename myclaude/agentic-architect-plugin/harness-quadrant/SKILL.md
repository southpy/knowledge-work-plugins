# Harness 四象限评估框架

## 框架核心

来自项目实战笔记的关键结论：**Harness（验收基线、执行边界、反馈信号、回退手段）往往比模型本身更决定系统能否收敛。** 高质量自动化验证和清晰目标缺一不可。

两个维度决定 Agent 系统的成败：

| 维度 | 定义 | 低分特征 | 高分特征 |
|------|------|----------|----------|
| **任务清晰度** (X轴) | 目标、边界、成功标准是否明确 | 需求模糊、多义、依赖大量隐式判断 | 目标单一、输入输出结构化、边界清晰 |
| **验证自动化程度** (Y轴) | 结果是否能被程序/系统自动检验 | 只能靠人工审查 | 单测/集成测试/规则引擎可自动判断对错 |

## 四象限定义

```
验证自动化程度
        高 │ Q2: 人力瓶颈区    │ Q1: ✅ Agent 黄金区
           │ 任务清晰但验收靠人  │ 目标明确+自动验证
           │ 吞吐上限=人工速度  │ Agent 最大发挥空间
           ├───────────────────┼───────────────────
        低 │ Q4: ❌ 双失效区   │ Q3: ⚠️ 方向失控区
           │ 两者皆缺          │ 自动化但目标模糊
           │ Agent 基本无效    │ 高效跑向错误方向
           └───────────────────┴───────────────────
                     低                    高
                              任务清晰度
```

### 各象限行动策略

**Q1（右上）— Agent 黄金区**
- Agent 完全发挥，优先构建
- 核心工作：工具链设计 + 幂等性 + 可观测性

**Q2（左上）— 人力瓶颈区**
- 先行动：把验收标准工程化（写测试、加 Linter、构建 CI）
- 不要先做 Agent，先把 Y 轴拉高到 Q1

**Q3（右下）— 方向失控区**
- 先行动：需求澄清会 + 写清 Acceptance Criteria
- 警告：此时做 Agent 会高效地跑向错误结果

**Q4（左下）— 双失效区**
- 先行动：需求探索 + 原型验证，不上 Agent
- Agent 在此区域是负投资

## 评估执行流程

每次评估按以下步骤执行：

### Step 1：提取场景要素

从用户描述中识别：
- **业务目标**：最终要实现什么结果
- **输入**：触发任务的数据/事件是什么
- **输出**：期望产出是什么格式/内容
- **验收方式**：目前怎么判断"做完了"且"做对了"

### Step 2：双维度打分（各 1-5 分）

**任务清晰度评分标准：**
- 5分：目标唯一、输入输出完全结构化、边界无歧义
- 4分：目标明确、少量边界案例需人工确认
- 3分：主流程清晰、但有多个分支需判断
- 2分：目标方向已知、但执行路径模糊
- 1分：连目标都有多种解释、高度依赖隐式知识

**验证自动化程度评分标准：**
- 5分：单测/集成测试完全覆盖、CI 自动通过即上线
- 4分：大部分可自动验证、少量需人工抽查
- 3分：有规则可验证核心逻辑、边界需人工
- 2分：只能靠人工 review、无结构化验收标准
- 1分：验收依赖主观判断、甚至没有明确验收标准

### Step 3：象限定位

- 清晰度 ≥ 3 且 自动化 ≥ 3 → Q1
- 清晰度 ≥ 3 且 自动化 < 3 → Q2
- 清晰度 < 3 且 自动化 ≥ 3 → Q3
- 清晰度 < 3 且 自动化 < 3 → Q4

### Step 4：输出评估报告

使用下方标准报告模板。

### Step 5：写入评估记录文件

将报告追加写入 `~/Documents/quadrant-assessments/YYYY-MM-DD.md`（若目录不存在则创建）。记录格式见下方。

## 标准报告模板

```markdown
## 🔷 Harness 四象限评估报告

**场景**：{一句话概括业务场景}
**评估时间**：{YYYY-MM-DD HH:MM}

### 维度评分

| 维度 | 得分 | 关键依据 |
|------|------|----------|
| 任务清晰度 | {X}/5 | {1-2句关键理由} |
| 验证自动化程度 | {Y}/5 | {1-2句关键理由} |

### 象限定位

**→ Q{N}：{象限名称}**

{2-3句描述当前场景在此象限的具体表现}

### 核心风险

{列出 1-3 个当前状态的主要风险，每条一句话}

### 行动建议

**立即优先级：**
1. {最高优先级行动}
2. {次优先级行动}

**Agent 投入建议：** {现在就做 / 先解决前置条件再做 / 暂缓}

**进入 Q1 的路径：**
- 清晰度缺口：{具体需要澄清的问题}
- 验证缺口：{具体需要构建的验收机制}
```

## 评估记录文件规范

文件路径：`~/Documents/quadrant-assessments/YYYY-MM-DD.md`

每次评估追加一条记录，格式：

```markdown
---
### [{HH:MM}] {场景简称}
- **象限**：Q{N} — {名称}
- **清晰度**：{X}/5 | **验证自动化**：{Y}/5
- **核心风险**：{最主要的1条风险}
- **行动**：{最高优先级行动}
---
```

执行写入时，用 Bash 检查目录是否存在并创建，追加内容到文件。如果文件不存在则创建并写入当天的标题 `# 四象限评估记录 — YYYY-MM-DD`。

---

## 反模式警示

评估时同步检查，若发现以下反模式，在报告末尾附注：

- **过早 Multi-Agent**：若 Q 值 < Q1 就提出多 Agent 方案
- **验证闭环缺失**：Agent 可以声称完成但无法验证
- **约束靠期望**：规则写在 prompt 里而非工具/hook 强制执行
- **目标模糊上自动化**：Q3 的典型陷阱

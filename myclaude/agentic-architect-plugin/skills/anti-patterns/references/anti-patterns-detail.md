# 8 条 Agentic 反模式 — 完整分析

---

## 反模式 1：系统提示当知识库

### 描述
System prompt 越来越长，把操作手册、领域知识、参考文档全都塞进去。结果是：关键约束被淹没，模型在长对话中选择性忽略末尾规则。

### 诊断信号
- System prompt > 1000 tokens
- 发现重复解释同一概念（说明之前的版本没起作用）
- Agent 在第 5 轮之后"忘记"早期设定的规则
- prompt 中有大段"背景介绍"或"参考信息"

### 根因
LLM 对长 prompt 存在注意力衰减。位于中间位置的内容被忽略的概率显著高于头尾。知识类内容天然适合按需加载，不适合常驻。

### 处置方法
1. **提取知识到 Skills**：把任何超过 3 行的"参考信息"移入独立 Skill
2. **常驻层只留约束**：身份定义 + 绝对禁止项 + 输出格式要求，目标 ≤500 tokens
3. **重复规则是信号**：同一规则出现 2 次说明之前写法不够强，合并为单一强约束

## 反模式 2：工具数量失控

### 描述
功能需求不断增加，工具集从 5 个增长到 20 个。Agent 面对大量相似工具时，选择准确率显著下降，出现幻觉调用（调用了不存在的参数组合）。

### 诊断信号
- 工具数量 > 10 个
- 存在功能相近的工具（`get_order` + `fetch_order` + `retrieve_order_by_id`）
- 调用日志中出现工具选择错误（选了功能相近但不完全匹配的工具）
- 存在调用频次为 0 的工具

### 根因
工具描述占用 token，10 个工具的描述可能消耗 2000+ tokens。更重要的是：相似工具描述对模型形成干扰，选择错误率随工具数量非线性增长。

### 处置方法
1. **合并重叠工具**：用可选参数替代多个单功能工具
2. **删除僵尸工具**：调用频次统计，0 调用率超过 2 周直接删除
3. **拆分 Agent 职责**：如果工具集覆盖 3 个以上业务域，考虑拆分专职子 Agent
4. **命名空间规范**：`orders__search`、`orders__get`，用前缀区分业务域

## 反模式 3：验证闭环缺失

### 描述
Agent 执行完任务后说"已完成"，但没有任何机制验证结果是否正确。这是最危险的反模式：系统在高置信度地做错误的事。

### 诊断信号
- 任务完成的唯一证据是 Agent 的文字声明
- 没有定义"什么算成功"
- 发现错误只能靠人工复查或下游系统报错
- 修改 prompt 后不知道是否改善了效果

### 根因
没有 Harness（验收基线）。Harness 的四要素：验收标准、执行边界、反馈信号、回退手段。缺少任何一项都会导致系统无法收敛。

### 处置方法
1. **为每类任务定义 Harness**：
   - 验收标准：结果满足什么条件算成功（可测量的）
   - 执行边界：允许重试几次，允许执行哪些操作
   - 反馈信号：从哪里获取成功/失败信号（不是 Agent 自己说）
   - 回退手段：失败时如何恢复到安全状态
2. **自动化验证**：单元测试、集成测试、或专门的 Validator Agent
3. **Pass@k 基准**：运行 k 次同一任务，统计成功率，建立基线后才上线

## 反模式 4：多 Agent 无边界

### 描述
引入多个 Agent 协作，但没有明确定义每个 Agent 的权限范围和状态所有权。结果：状态被多个 Agent 并发修改导致漂移，出错后无法归因（是哪个 Agent 做错了？）。

### 诊断信号
- 多个 Agent 都可以修改同一份数据/状态
- 故障后需要人工逐一检查每个 Agent 的日志才能定位问题
- 不同 Agent 的决策相互矛盾
- Agent 之间通过共享内存/全局变量通信

### 根因
多 Agent 系统本质是分布式系统，分布式系统的经典问题（并发、一致性、故障隔离）在这里全部重演。

### 处置方法
1. **单一状态所有者**：每份数据只能由一个 Agent 修改，其他 Agent 只读
2. **Worktree 隔离**：使用 Git worktree 为每个 Agent 提供独立工作空间
3. **明确角色权限矩阵**：
   ```
   | Agent        | 读取 | 写入 | 删除 |
   |--------------|------|------|------|
   | planner      | ✅   | ✅   | ❌   |
   | executor     | ✅   | ✅   | ❌   |
   | reviewer     | ✅   | ❌   | ❌   |
   ```
4. **消息传递替代共享状态**：Agent 间通过结构化消息通信，不共享内存

## 反模式 5：记忆不整合

### 描述
对话超过 20 轮后，Agent 的决策质量明显下降。原因是上下文窗口接近上限，早期的重要信息被截断，Agent 开始基于不完整的信息做决策。

### 诊断信号
- 长对话中 Agent 开始重复提问已经回答过的问题
- 对话超过 30 轮后出现前后矛盾的决策
- Token 使用接近模型上下文窗口上限（>80%）
- 没有跨会话记忆（每次新对话都从零开始）

### 根因
上下文窗口是有限资源，没有主动管理等于任其耗尽。记忆整合（压缩 + 外化）是长任务稳定运行的前提。

### 处置方法
1. **设置 Token 监控阈值**：超过 70% 触发警告，超过 80% 触发自动压缩
2. **滑动窗口策略**：保留最近 N 轮完整对话，更早的内容用摘要替代
3. **状态外化**：任务状态写入文件系统，不依赖上下文窗口传递
4. **MEMORY.md 整合**：跨会话的关键决策、约定、错误教训写入 MEMORY.md

## 反模式 6：没有评测

### 描述
对 Agent 进行了修改（prompt 调整、工具变更、模型升级），但不知道修改是否改善了效果，也不知道是否引入了回归。

### 诊断信号
- 没有 eval 数据集
- 修改 prompt 后只靠"感觉好多了"判断效果
- 用户反馈的 bug 在内部没有对应测试用例
- 不同人对 Agent 质量的评价不一致（没有客观基准）

### 根因
没有评测等于没有方向感。在没有可靠反馈的情况下对系统做修改，是在随机游走。

### 处置方法
1. **失败案例立刻转测试用例**：每个用户报告的 bug → 对应 eval case
2. **Pass@k vs Pass^k**：
   - Pass@k（k 次中至少 1 次成功）：验证能力边界，探索上限
   - Pass^k（k 次全部成功）：验证上线质量，保证稳定性
3. **先修评测再动 Agent**：评测系统本身有问题时，不要基于失真信号调整方向
4. **自动化评测流水线**：每次代码提交触发 eval 运行，回归立即可见

## 反模式 7：过早引入多 Agent

### 描述
任务稍微复杂一点就引入多 Agent 架构，结果是协调开销（消息传递、状态同步、错误传播）远大于并行带来的收益。

### 诊断信号
- 多 Agent 架构的总体延迟高于单 Agent 处理同样任务
- 大量时间花在 Agent 间的协调消息上
- 子 Agent 的职责和主 Agent 高度重叠
- 调试需要追踪多个 Agent 的调用链

### 根因
多 Agent 是解决单 Agent 能力瓶颈的方案，不是提升性能的默认选择。在单 Agent 能力上限还没摸清之前引入多 Agent，是复杂性的过早投入。

### 处置方法
1. **先压榨单 Agent 上限**：通过更好的工具设计、更精准的 prompt、更好的上下文管理，先把单 Agent 能力推到极限
2. **明确引入多 Agent 的原因**：并行处理（速度）、专业分工（质量）、隔离故障（可靠性）— 必须是其中之一
3. **任务图先于 Agent 图**：先画出任务的依赖关系（DAG），再决定哪些节点需要独立 Agent
4. **子 Agent 只回传摘要**：搜索和调试细节留在子 Agent 上下文，主 Agent 只接收结构化结果

## 反模式 8：约束靠期望不靠机制

### 描述
关键规则写在 prompt 里，Agent 有时遵守有时不遵守。每次出问题都是"这次 Agent 没按照规则来"。这不是 Agent 的问题，是架构设计的问题。

### 诊断信号
- 同一条规则被违反超过 1 次
- 规则执行结果不一致（相同输入，不同 Agent 运行产生不同合规性）
- 通过在 prompt 里加粗、加星号、写"务必"来强调规则（说明 LLM 在忽略它）
- 规则违反只能在事后发现

### 根因
LLM 是概率性系统，prompt 是软约束。对于不可违反的规则，必须用确定性机制执行。

### 处置方法
1. **规则分级**：
   - 软规则（偏好）：prompt 合适
   - 硬规则（必须）：Tool 验证 + Hook 拦截
2. **PreToolUse Hook**：在 Agent 执行写操作前强制校验
3. **输出 Validator**：Agent 生成结果后，自动运行验证器检查合规性
4. **Linter/Schema 验证**：结构化输出用 JSON Schema 验证，不靠 Agent 自觉

```python
# 正确做法：约束在代码层执行
def pre_write_hook(file_path: str, content: str) -> ValidationResult:
    if not file_path.endswith('.py'):
        return ValidationResult(block=True, reason="只允许写入 .py 文件")
    if 'eval(' in content:
        return ValidationResult(block=True, reason="禁止使用 eval()")
    return ValidationResult(block=False)
```
